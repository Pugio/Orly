## Part 1: Bugs & Correctness Fixes

Real bugs that affect projector output. Fix before filming.

### Bug 1: Projector uses bounding-box resize, not perspective warp

**File:** `client/overlay_manager.py` → `place_on_canvas()`

**Problem:** The projector path transforms placement corners through `H_proj`, computes an axis-aligned bounding box, and `cv2.resize()`s into it. This discards the perspective correction — off-axis projector setups show visible drift on large overlays.

**Fix:** Use `cv2.getPerspectiveTransform()` + `cv2.warpPerspective()` to warp overlay content into the destination quad.

```python
# Instead of resize into bounding box:
src_corners = np.array([[0, 0], [w, 0], [w, h], [0, h]], dtype=np.float32)
M = cv2.getPerspectiveTransform(src_corners, dst_corners)
warped = cv2.warpPerspective(overlay, M, (self.proj_width, self.proj_height))
mask = warped.sum(axis=2) > 0
canvas[mask] = warped[mask]
```

### Bug 2: Calibration (x,y) vs rendering (y,x) axis order mismatch

**Files:** `calibration/projector_calibrate.py`, `client/overlay_manager.py`

**Problem:** Calibration defines points as `(x, y)`. Overlay manager feeds `[y, x]` into `perspectiveTransform`. With a real calibration `H_proj`, overlays land transposed.

**Fix:** Swap columns before `perspectiveTransform`:
```python
corners_xy = src_corners[:, :, ::-1]  # [y,x] → [x,y] to match calibration
dst_corners = cv2.perspectiveTransform(corners_xy, self.H_proj)
```

**Tests (`tests/test_axis_convention.py`):**

```

test_top_left_placement_lands_top_left
test_bottom_right_placement_lands_bottom_right
```

**Fix:** One line in `handle_tool_result()`:

## Part 2: Renderer Registry

### Motivation

Adding a new overlay type currently means touching 5 files:

1. New renderer in `client/renderer/X.py`
2. Add to `_VALID_TYPES` in `backend/tools.py`
3. Add `elif` branch + field extraction in `overlay_manager.py`
4. Add data format description to `project_overlay()` docstring
5. Add usage prose to system prompt in `agent.py`

A hackathon goal is showing others can hack on this. So: **one file to add a renderer, zero other files to touch.**

### Design

Each renderer module registers itself by exporting a `SPEC` dict. A central registry collects them. The overlay manager dispatches from the registry. The tool docstring and system prompt content-type docs are generated from specs.

**The spec is a plain dict.** No base classes, no metaclasses, no decorators.

```python
# client/renderer/number_line.py

def render_number_line(data: dict, width: int, height: int, title: str = "") -> np.ndarray:
    """Render a number line with labeled points and shaded ranges."""
    min_val = data.get("min_val", 0)
    max_val = data.get("max_val", 10)
    points = data.get("points", [])
    ranges = data.get("ranges", [])
    # ... rendering code ...

SPEC = {
    "name": "number_line",
    "description": "A number line with labeled points and shaded ranges.",
    "data_format": (
        'For "number_line": {"min_val": -5, "max_val": 5, '
        '"points": [{"value": 2, "label": "x", "color": "#00ff00"}], '
        '"ranges": [{"start": -1, "end": 3, "color": "#ffff00", "label": "solution set"}]}.'
    ),
    "prompt_hint": "Use for arithmetic, inequalities, fractions, or any 1D numeric visualization.",
    "render": render_number_line,
}
```

**The render function signature is uniform:** `(data: dict, width: int, height: int, title: str = "") -> np.ndarray`. Every renderer receives the raw `data` dict and extracts what it needs. This eliminates the per-type field extraction in the overlay manager.

**Registry (`client/renderer/registry.py`):**

```python
"""Renderer registry — auto-discovers SPEC dicts from renderer modules."""

import importlib
import pkgutil
from pathlib import Path

_REGISTRY: dict[str, dict] = {}


def _discover():
    """Import all modules in client/renderer/ and collect SPEC dicts."""
    package_dir = Path(__file__).parent
    for info in pkgutil.iter_modules([str(package_dir)]):
        if info.name.startswith("_") or info.name == "registry":
            continue
        mod = importlib.import_module(f"client.renderer.{info.name}")
        spec = getattr(mod, "SPEC", None)
        if spec and "name" in spec and "render" in spec:
            _REGISTRY[spec["name"]] = spec


def get(name: str) -> dict | None:
    """Get a renderer spec by content_type name."""
    if not _REGISTRY:
        _discover()
    return _REGISTRY.get(name)


def all_specs() -> list[dict]:
    """Get all registered specs (sorted by name)."""
    if not _REGISTRY:
        _discover()
    return sorted(_REGISTRY.values(), key=lambda s: s["name"])


def valid_types() -> set[str]:
    """Get the set of all registered content_type names."""
    if not _REGISTRY:
        _discover()
    return set(_REGISTRY.keys())


def data_format_docs() -> str:
    """Generate the data format documentation for the project_overlay docstring."""
    lines = []
    for spec in all_specs():
        lines.append(f'For "{spec["name"]}": {spec["data_format"]}')
    return "\n              ".join(lines)


def prompt_overlay_docs() -> str:
    """Generate the OVERLAY CONTENT TYPES section for the system prompt."""
    lines = []
    for spec in all_specs():
        hint = spec.get("prompt_hint", "")
        lines.append(f'- "{spec["name"]}": {spec["description"]} {hint}')
    return "\n".join(lines)
```

**Overlay manager change — replace the entire elif chain:**

```python
def render_overlay(self, content_type, placement, title, data):
    y_min, x_min, y_max, x_max = placement
    overlay_w = max(1, int((x_max - x_min) / 1000.0 * self.proj_width))
    overlay_h = max(1, int((y_max - y_min) / 1000.0 * self.proj_height))

    from client.renderer.registry import get
    spec = get(content_type)
    if spec:
        return spec["render"](data, overlay_w, overlay_h, title=title)

    # Unknown type: black image
    return np.zeros((overlay_h, overlay_w, 3), dtype=np.uint8)
```

**Backend tools change — generate `_VALID_TYPES` from registry:**

```python
# backend/tools.py
from client.renderer.registry import valid_types
_VALID_TYPES = valid_types()
```

**Backend agent change — generate docstring and prompt sections:**

```python

# In SYSTEM_PROMPT, the OVERLAY CONTENT TYPES section:

# In project_overlay docstring, the data format section:

```

### Migrating existing renderers

Each existing renderer gets a `SPEC` added at module level. The render function signature is normalized to `(data, width, height, title="")`. Example migration for graph:

```python
# client/renderer/graph.py — before:
def render_graph(expression, x_range, y_range, width, height): ...

# after:
def _render_graph_impl(expression, x_range, y_range, width, height): ...
    # existing code unchanged

def render_graph(data: dict, width: int, height: int, title: str = "") -> np.ndarray:
    return _render_graph_impl(
        data.get("expression", "x"),
        data.get("x_range", [-10, 10]),
        data.get("y_range", [-10, 10]),
        width, height,
    )

SPEC = {
    "name": "graph",
    "description": "A mathematical function plot y=f(x).",
    "data_format": '{"expression": "x**2 - 3*x + 2", "x_range": [-5, 5], "y_range": [-5, 10]}.',
    "prompt_hint": "Use ONLY for plotting a single function y=f(x).",
    "render": render_graph,
}
```

The old function name is preserved as `_render_graph_impl` so existing direct callers (if any) can be updated incrementally. Tests that call `render_graph(expression, x_range, y_range, w, h)` directly would call `_render_graph_impl` instead, or use the new `render_graph(data, w, h)` signature.

### Special cases

**`highlight`** returns BGRA. Its spec's render function handles the conversion internally:
```python
def render_highlight(data, width, height, title=""):
    bgra = _render_highlight_impl(width, height, data.get("color", "#00ffff"))
    return bgra[:, :, :3]
```

**`image`** is async (background thread). It's still registered in the registry for validation and prompt docs, but `OverlayManager.handle_tool_result()` still special-cases it for the async loading flow. The registry handles the common case; the manager handles the exception.

### What this gives you

**To add a new renderer (e.g. "timeline"):**

1. Create `client/renderer/timeline.py`
2. Write `render_timeline(data, width, height, title="") -> np.ndarray`
3. Add `SPEC = {"name": "timeline", "description": "...", "data_format": "...", "prompt_hint": "...", "render": render_timeline}`
4. Done. No other files to touch.

The registry auto-discovers it. The tool validates it. The overlay manager renders it. The system prompt documents it.

### Implementation Steps

1. Create `client/renderer/registry.py` with discovery + query functions.
2. Normalize all existing renderer signatures to `(data, width, height, title="")`.
3. Add `SPEC` to each existing renderer module (graph, annotation, highlight, markdown, number_line, geometry, chemistry, steps).
4. Replace the elif chain in `overlay_manager.py` with registry lookup.
5. Replace `_VALID_TYPES` in `tools.py` with `valid_types()`.
6. Generate data format docs and prompt overlay section from registry.
7. Keep `image` special-cased in `handle_tool_result()` for async flow.

## Part 3: Performance Fixes

### Perf 1: Eliminate JPEG round-trip in video hot path

**File:** `client/main.py`, `client/camera.py`

**Problem:** Video loop encodes rectified frame to JPEG, then decodes the same JPEG for trackers/programs. Wastes CPU, degrades quality.

**Fix:** `get_rectified_frame()` returns `(jpeg_bytes, raw_ndarray, H_cam)`. JPEG goes to backend, raw ndarray goes to trackers.

**Tests:**
```
test_camera_returns_raw_frame — returns 3-tuple with ndarray
test_video_loop_passes_raw_to_tracker — tracker gets ndarray not decoded JPEG
```

### Perf 2: Move tracker/program work off transport loop

**File:** `client/main.py`

**Problem:** Tracker updates and `on_frame()` callbacks run inline in the asyncio video loop. Slow tracking blocks frame sending.

**Fix:** Offload to thread pool:
```python
loop.run_in_executor(None, program_runtime.process_frame, raw_frame)
```

**Tests:**
```
test_slow_tracker_does_not_delay_frame_sending
```

### Feature 6: Hand/Pointing Gesture Detection

**What:** Student points at a problem; Lumi identifies what's there.

**New file: `client/hand_detector.py`** — three testable pure functions + a tracker:

```python
@dataclass
class PointingResult:
    fingertip: tuple[float, float]  # (y, x) in 0-1000
    confidence: float
    is_pointing: bool

def is_pointing_gesture(landmarks, extension_threshold=0.1) -> bool:
def fingertip_to_table_coords(pixel_y, pixel_x, frame_h, frame_w) -> tuple[float, float]:

@dataclass
class PointingTracker:
    stability_time: float = 1.0   # seconds at same position to confirm
    jitter_threshold: float = 50.0  # max position drift in 0-1000 units
    debounce: float = 2.0          # min seconds between notifications
```

**Integration:** Run on each video frame. Stable point (>1s) → notify backend. Project small bright dot at fingertip.

**System prompt:** `POINTING` section — treat pointing notifications as "this one."

**Tests (`tests/test_hand_detector.py`):** ~16 — gesture classification, coord conversion, tracker debounce/stability.
